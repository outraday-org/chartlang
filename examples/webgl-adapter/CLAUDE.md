# examples/webgl-adapter/

Example adapter — **not published to npm**. Renders chartlang emissions to a
**raw WebGL2** GPU renderer with **zero npm chart-lib dependency** (like the
canvas2d reference). Mirrors `examples/canvas2d-adapter/` in shape (factory +
loop + capabilities-only conformance default export).

> **Current scope: FULL PARITY (through Task 14).** The package scaffold,
> `WEBGL_CAPABILITIES`, the GL infra (`src/webgl/*`), the pure
> state→descriptor bridge (`state.ts` / `ingest.ts` / `buildFrame.ts` /
> `layer-descriptor.ts`), the per-pane `Renderer`, the REAL
> `createWebglAdapter` factory + `runWebglLoop`, and every GPU program arm
> (candle bodies/wicks, line strips, vertical bars, filled bands, cursors,
> markers) + the axes / overlay / interaction / drawings layers all exist.
> Task 14 closed parity: the candle / background overrides, the right-edge
> volume-profile, and the per-pane z-order render pass (the SHARED
> `sortByRenderOrder`). A mounted canvas renders the full surface; headless
> stays a safe no-op. The per-task sections below remain the orientation map;
> see the "Overrides + z-order pass + volume-profile (Task 14)" section for
> the parity-completing layer.

## Conventions

- `package.json` carries `"private": true` and the unscoped name
  `chartlang-example-webgl-adapter`. No `publishConfig`. No npm chart-lib
  dependency (raw WebGL2). Same six-file §22.4 surface generated by
  `pnpm scaffold` — do not hand-edit the template files; edit
  `scripts/scaffold.ts` and re-run.
- The intent is "copy from this folder when writing your own GPU adapter".
- **No cross-example src imports.** The shared geometry / interaction /
  glyph / canvas-sink surface is consumed ONLY through the public
  `@invinite-org/chartlang-adapter-kit` (+ `/canvas`) and
  `@invinite-org/chartlang-host-worker` boundaries — never another
  example's `src/`.

## Invariants

- **The default export is capabilities-only.** `DEFAULT_ADAPTER`
  (`src/defaultAdapter.ts`, `id: "webgl-reference-default"`) is the headless
  conformance export: the conformance runner reads `capabilities` only, so
  `candles` / `onEmissions` / `dispose` are no-ops — no GL context or Worker
  host is spun up for an emission-contract test. Its id is distinct from the
  registry id (`"webgl"`) and the production factory's adapter id
  (`"webgl-reference"`). Production uses `createWebglAdapter`.

- **Full-surface adapter — conformance is gated via `pnpm conformance`
  (`scripts/run-conformance.ts`), NOT a per-package `src/conformance.test.ts`.**
  webgl claims canvas2d's full ~256-scenario surface, and that suite under v8
  coverage exceeds the root `pnpm test` (`vitest run --coverage`) per-test
  timeout — so it does NOT carry a per-package conformance test (and therefore
  no `@invinite-org/chartlang-conformance` devDep). This mirrors the canvas2d
  reference adapter exactly: the four NARROW library adapters (echarts / konva /
  uplot / lwc, ~23-token surface) keep the cheap per-package test, but a
  full-surface adapter is run for pass/fail only by `pnpm conformance` (webgl
  is in the registry, so it is iterated there). Do NOT add a per-package
  `conformance.test.ts` back.

- **`WEBGL_CAPABILITIES` declares the FULL surface, byte-for-byte the same
  shape as `CANVAS2D_CAPABILITIES`** (the full Phase-5 plot inventory + all
  63 drawing kinds + alerts + alert conditions + logs + canonical timeframe
  metadata + the unlimited sub-pane sentinel + `syminfo.*` + multi-timeframe
  / multi-symbol). Declaring the whole set NOW (even though the renderer
  lands later) keeps the adapter interchangeable with the other five and
  exercises the entire conformance surface.

- **The factory is headless-constructible from `{ width, height }` alone.**
  `createWebglAdapter` does NOT call `canvas.getContext("webgl2")` and never
  touches a real GL context unless a real `canvas` or the `gl`
  (`WebGL2RenderingContext`) test seam is supplied. With neither, no
  `Renderer` is built and `onEmissions` runs ingestion only (a safe no-op
  draw). Headless tests pass a bare `{ width, height }` + an in-process stub
  `host`. See "Renderer + factory wiring (Task 5)" for the resolution
  precedence + dispose ownership.

- **`runWebglLoop` mirrors canvas2d's `runRendererLoop` shape** (iterate
  `candles({interval})` → mirror into the bar window → `await host.push` →
  yield once → `host.drain` → `onEmissions`; honour `opts.signal` with a
  silent return on abort). Renderer state lives off the public handle via a
  module-local `WeakMap`; a `runWebglLoop` call on a foreign handle throws
  the documented sentinel.

## GL infra (Task 2) — `src/webgl/`

The GPU-foundation primitives Tasks 3–14 build on. Each file carries the
4-line invinite provenance header (SHA `cd883292`). Browser-only `gl.*`
driving lands later; the pure parts are node-unit-tested here.

- **`gl-context.ts` has the SINGLE device-px rounding + clamp site —
  `resolveBackbufferSize(cssW, cssH, dpr)`** (pure, node-tested). It
  rounds `cssPx × dpr` to integer backbuffer dimensions and clamps each
  axis to `MAX_CANVAS_PX = 16384`, returning an `exceeded` flag.
  `createGlContext().resize` is the ONLY caller; on `exceeded` it throws
  (a runaway-resize-loop is a bug). Do not add a second rounding site.
- **`CONTEXT_OPTIONS` is the frozen MSAA bag** `{ alpha, antialias,
  premultipliedAlpha, preserveDrawingBuffer:false }` — `antialias:true`
  is the smooth-look contract; a test pins the exact bag.
- **`dispose()` resets `canvas.{width,height}` to the HTML default
  (300×150) and does NOT call `WEBGL_lose_context.loseContext()`.** A
  same-canvas remount (StrictMode / route swap) re-uses the existing
  context; flipping it lost makes every later `gl.createShader` return
  null. GL resources are released per-program / per-buffer in later
  tasks' `dispose`. invinite's `tcLog` / DEV resource-tracker proxy /
  rerender-storm channel / double-DPR forensic guards were dropped
  ("translate, not transcribe").
- **`program-cache.ts` is the GENERIC `getProgram(gl, key, factory)` +
  `clearProgramCache(gl)` form, NOT invinite's typed program record.**
  invinite's record names ten concrete program classes that land in
  Tasks 6–13; the ported contract is the per-`gl` lazy lifetime
  (`WeakMap<gl, Map<key, instance>>`: same instance per `(gl, key)`,
  fresh per gl, cleared on dispose), not the program-set coupling. Each
  later task registers its program under a string key.
- **`vao.ts` uses a local structural `VaoBuffer = { readonly glBuffer:
  WebGLBuffer }` seam in place of Task 3's `PooledBuffer`.** Task 3's
  `BufferPool.PooledBuffer` structurally satisfies it, so the pool is
  passed to `new Vao(gl, layouts)` with NO edit here — this keeps Task 2
  free of a forward import on the unwritten buffer-pool module. The
  `attribute.location < 0` skip + the `divisor > 0` instancing branch are
  preserved.
- **`geometry.ts` exports the two static triangle-strip quads verbatim**
  (`UNIT_QUAD_TRIANGLE_STRIP`, `Y_ZERO_QUAD_TRIANGLE_STRIP`). Read-only
  by convention (a `Float32Array` view is not freezable in V8).
- **Tests are node-safe.** Pure helpers (`resolveBackbufferSize`,
  `CONTEXT_OPTIONS`, the quads) and the stub-`gl` call-sequence tests for
  `Program` / `Vao` / `getProgram` run under the node root vitest. The
  DOM-driven `createGlContext` resize/dispose paths are browser-only and
  NOT node-tested (this adapter is not coverage-gated).

## Pure math + buffers (Task 3) — `src/webgl/`

The pure math + GPU-memory layers behind the Task-2 surface. Each file
carries the 4-line invinite provenance header (SHA `cd883292`); the
shader-module files additionally note the luma.gl/deck.gl pattern
inspiration (re-implemented in-tree, NOT an npm dep). All five layers are
pure (math / GLSL strings) or stub-`gl` book-keeping → node-unit-tested.

- **`projection.ts` — `ortho2d(left, right, bottom, top)` → column-major
  `Float32Array(9)` mat3.** Layout is `[sx, 0, 0,  0, sy, 0,  tx, ty, 1]`
  (consumed by `mat3 uProj`, fed via `Program.setUniformMatrix3fv`). The
  adapter feeds it the world window resolved by the shared `ViewController`
  (Task 4), NOT a reimplementation of the window math. **Divergence from
  the pinned source:** a degenerate span (`left === right` / `bottom ===
  top`) is clamped to `EPSILON = 1e-9` before the divide so no entry is
  `Infinity`/`NaN` (invinite divided by zero) — the same defensive choice
  the canvas2d viewport makes. A unit test pins finite output for both
  axes.
- **`viewport.ts` — `paneViewport(pane, canvasCssHeight, dpr)` → `DeviceRect`
  is the SINGLE CSS-px → device-px rounding site.** Each edge passes
  through one `Math.round(edge × dpr)`; width/height are the difference of
  two rounded edges, so adjacent panes share a tile-tight integer seam even
  at non-integer `dpr`. Top-left CSS origin in, bottom-left device origin
  out. Do not add a second rounding site (mirrors `gl-context.ts`'s single
  `resolveBackbufferSize` site).
- **`shader-modules/assemble.ts` is the assembly contract.**
  `assembleVertexShader`/`assembleFragmentShader({ modules, body })`
  concatenate `#version 300 es` + precision (`highp` vs `mediump`) +
  modules (in order) + body. Module order is load-bearing: a later module
  may reference helpers from an earlier one, so consumers list modules in
  dependency order. `project32` (`worldToSnappedNdc` + `dojiInflateNdcY`,
  + the `uViewportSize`/`uDpr` uniform tuple `PROJECT32_UNIFORMS`),
  `nan-skip` (segment/neighbor NaN predicates), and `aa` (`disk_aa_alpha`
  fwidth/smoothstep feather) are GLSL **strings** — tested by asserting the
  exported symbol names, not by GL execution.
- **`buffer-pool.ts` lifetime / dirty-range / consumer-registry policy.**
  - `PooledBuffer` STRUCTURALLY satisfies Task-2's `VaoBuffer` (it is
    `VaoBuffer & { id, cpu, capacity, length, uploadedLength,
    lastUploadedFrame }`), so a pool slot feeds `new Vao(gl, layouts)`
    with no edit to `vao.ts`.
  - **Capacity grows to the next power of two; `cpu` identity is preserved
    while capacity suffices** (callers may cache the `cpu` reference across
    frames as long as they re-acquire each frame). A grow resets
    `uploadedLength = 0` so the next `upload` re-runs a full `bufferData`.
  - **`upload` routing:** full `bufferData` when `hint === undefined` OR
    `uploadedLength === 0` OR `uploadedLength !== length` (the size-change
    safety net); clean-skip (no GL call) when `hint.dirtyLength === 0`;
    partial `bufferSubData` over the dirty range otherwise. Dirty-range
    values are floats; the pool converts the byte stride.
  - **Same-frame upload-once gate:** `beginRendererFrame()` bumps a
    module-scope `CURRENT_FRAME`; a SHARED non-`scratch` slot whose
    `lastUploadedFrame === CURRENT_FRAME` short-circuits (the content-key
    contract guarantees byte-identical data). `scratch: true` slots
    (heterogeneous multiplexed buffers) opt OUT and always upload.
  - **Consumer registry:** `acquire(id, floats, consumerKey)` records the
    consumer in a per-slot set; the GL buffer frees only on the LAST
    release (`release(id, key)` per-id, `disposeByConsumer(key)` across
    all slots, `disposeByPrefix(prefix)` for full-symbol cleanup). N panes
    sharing a content-key collapse to one buffer / one upload.
  - **Dropped from the pinned source** ("translate, not transcribe"): the
    `tcLog` observability hook and the `import.meta.env.DEV`-gated GPU-byte
    counter (`readAndResetUploadedBytes`) — `import.meta.env` is Vite-only
    (undefined under node vitest) and the counter is bench-only. The frame
    counter is KEPT (it is the correctness contract for the gate). The
    README §9 tail-mutation *perf* fast-path is the `bufferSubData`
    dirty-range routing, which is ported here for correctness; only the
    DEV byte-accounting is deferred/dropped.
- **Tests are node-safe.** `projection`/`viewport`/`assemble`/`project32`/
  `nan-skip`/`aa` are pure; `buffer-pool` + `buffer-pool.shared` run
  against a recording stub `gl` (bare object `as unknown as
  WebGL2RenderingContext`, `vi.fn()` recorders) — the same idiom as the
  Task-2 `program`/`vao`/`program-cache` tests.

## State + emission ingestion + descriptors (Task 4) — `src/`

The pure, headless bridge between chartlang emissions and the GPU — NO `gl`
in any of these files. `state.ts` / `ingest.ts` / `buildFrame.ts` are
chartlang-native (2-line MIT header); `layer-descriptor.ts` (+ its test)
carries the 4-line invinite provenance header (SHA `cd883292`) — its
descriptor SHAPE + color resolvers are the port, the React/perf/indicator
coupling was dropped ("translate, not transcribe"). Task 4 lands the pure
modules + unit tests ONLY; Task 5 wires `applyEmissions` → `buildFrame` →
GL paint into `createWebglAdapter.onEmissions` and feeds each pane window
to `ortho2d`.

- **`AdapterState` (`state.ts`) mirrors the canvas2d reference's stores
  byte-for-concept** (`bars`, `paneOrder` [`"overlay"` always index 0],
  `plotSeries` + `plotSeriesStyle` keyed `${paneKey}|${slotId}`,
  `plotOverlays` keyed `${slotId}@${time}`, `hlines` keyed by slot id,
  `recentAlerts`/`currentAlertConditions`/`recentLogs`, `drawings`, `seq`,
  `overlaySeq`/`drawingSeq`, `palette`, `view`, `initialVisibleBars?`) MINUS
  the canvas2d render handles (`ctx`/`dpr`/interaction detach — GL / loop
  concerns Task 5 owns). `createAdapterState(opts?)` builds `view` via the
  shared `createViewController()` and seeds `palette` from `opts.palette ??
  DEFAULT_PALETTE`. `resetAdapterState` is the `dispose` helper (clears
  stores, retains `view` + `palette` for a same-canvas remount).
- **The adapter-kit ViewController / yRangeInWindow consumption seam is the
  central reuse contract — NEVER fork it.** `buildFrame` resolves the
  x-window via `state.view.resolveXWindow(dataXMin, dataXMax,
  autoFollowXMin)` and the y-window via `yRangeInWindow(candidates, win)`
  (both imported from `@invinite-org/chartlang-adapter-kit`). The
  `initialVisibleBars` contract is canvas2d's exact one: `autoFollowXMin =
  bars[len - N].time` only when `N` is set, `> 0`, and `len > N` (else
  `undefined` ⇒ fit all data). The `(0,1)` empty-bars fallback, the
  degenerate-`yMin===yMax` ±1 widen, and the ±5% (`Y_AXIS_PADDING`) pad all
  mirror canvas2d's `computePaneViewport`. hlines fold into the y-range by
  the caller (`yRangeInWindow` excludes no-`x` rows).
- **`buildFrame` stays in WORLD space (time, price) — it is ignorant of clip
  space.** Each `PaneRenderState.window` is the rectangle Task 5 feeds to
  `ortho2d`; descriptors carry world coordinates, not pixels. **xShift
  therefore uses the world-space `shiftedBarTime` (world time out), NOT
  `projectShiftedX` (pixel out)** — `extendXMaxForShifts` widens `dataXMax`
  for a `+k` future series shift the same way canvas2d does, but in world
  time. (Drawing / glyph future-anchor widening is deferred to Tasks 12–14;
  their descriptors are not built here yet.)
- **The emission→descriptor mapping contract (`layer-descriptor.ts`):** the
  `LayerDescriptor` union carries ONLY plain data (`Float32Array` packs in
  world units + unit `RgbaUnit` colors) — no `gl`, no bar-index frames, no
  dirty-range / scale-group / `_lineStyle` perf fields (dropped from the
  invinite port). `buildFrame` builds the MVP descriptors the Task-4 state
  can express: `candle-bodies` + `candle-wicks` (overlay only, packed
  `[x,open,high,low,close,isBull]` / `[x,low,high,isBull]`) and `line-strip`
  per line / step-line series, ordered by each series' most-recent
  `(z, seq)`. A `value:null` OR `colorValue:null` gap packs `NaN` y so the
  Task-7 program skips the segment. `vertical-bars` / `filled-band` /
  `cursor` / `marker` / `drawing` / `text` are TYPED now but built by Tasks
  10–13 (histogram / area / filled-band series styles are accumulated by
  `applyEmissions` but not yet descriptor-built — Tasks 10–11).
- **Palette / colors resolution.** `Palette` + `DEFAULT_PALETTE` are defined
  LOCALLY in `layer-descriptor.ts` (the no-cross-example-`src`-import
  invariant forbids importing canvas2d's), mirroring canvas2d's shape +
  hexes (the invinite `#26a69a`/`#ef5350` bull/bear). `hexToRgbaUnit(hex,
  alpha)` parses `#rgb`/`#rrggbb`/`#rrggbbaa` → `[0,1]` RGBA (a malformed
  string falls back to black, never `NaN`). `isBullish(bar)` (close ≥ open,
  doji bullish) drives bull/bear selection. `resolvePaintColor(colorValue,
  static, fallback)` is the line-family 3-state precedence (omitted ⇒
  `static ?? fallback`; present ⇒ override; `null` ⇒ `null` gap). **The
  Task-1 `opts.palette` opt is NOT yet on `CreateWebglAdapterOpts`** — Task 5
  adds it + threads it into `createAdapterState`.
- **`visible: false` drops the mark and excludes it from the y-scale —
  self-scaled "hidden by omission" semantics (canvas2d parity).** `ingest.ts`'s
  `applyPlot` early-returns on `plot.visible === false` before storing any
  series point / hline / overlay, so a hidden slot contributes nothing the
  GPU draws AND nothing to `buildFrame`'s `computeYRange` (it never stretches
  the pane scale). The renderer re-derives every frame (no persistent legend),
  so omitting the mark IS "keep the slot hidden" — re-enabling re-emits it. The
  runtime writes `visible: false` for BOTH authoring (`plot(value, { visible
  })`) and host overrides; the same guard covers both. Orthogonal to
  `value: null` (a per-bar gap) — see the adapter-kit `PlotEmission.visible`
  normative contract. Covered by `ingest.test.ts`.
- **Pure → headlessly unit-tested (§16.3).** `layer-descriptor.test.ts`
  (colors/palette), `ingest.test.ts` (accumulation), `buildFrame.test.ts`
  (window/y-fit/descriptors). The window/y-fit math is already tested in
  adapter-kit, so these tests pin THIS adapter's mapping, not the shared
  helpers. No `gl`, node-safe.

## Renderer + factory wiring (Task 5) — `src/webgl/Renderer.ts` + `createWebglAdapter.ts`

End-to-end now: `onEmissions` → `applyEmissions` (Task 4) → `buildFrame`
(Task 4) → `renderer.update(panes)` + `scheduleDraw()`. The GPU program arms
(candle bodies/wicks, line strips, …) land in Tasks 6–13 behind the
renderer's dispatch seam; until then a real canvas clears per pane but draws
no geometry.

- **`Renderer` (ported, provenance header) is the per-pane frame
  orchestrator.** `new Renderer(glContext)`; `update(panes)` stages a
  snapshot; `scheduleDraw()` coalesces via `requestAnimationFrame` (only the
  latest snapshot paints); `drawNow()` is the synchronous paint (the path
  node tests drive — rAF is browser-only and `/* v8 ignore */`d);
  `dispose()` is idempotent (cancels the pending rAF, disposes only the
  `usedPrograms` it dispatched through, swallows a lost-context throw via
  `options.onError`). invinite's React/bus coupling + DEV stats / forensics
  (`getStats`, paneCount/fps/layer-build windows, `tcLog`) were dropped
  ("translate, not transcribe").
- **`drawNow` clears with the X-3 scissor-clear order, INLINED.** invinite's
  `frame-state.beginFrame` was NOT ported (Tasks 1–4 skipped it), so the
  one helper lives as `Renderer.beginFrame()`: `disable(SCISSOR_TEST)` →
  full-canvas `clear(COLOR_BUFFER_BIT)` → `enable(SCISSOR_TEST)` +
  `enable(BLEND)` + premultiplied-alpha `blendFuncSeparate`. A prior frame
  may leave SCISSOR_TEST enabled with a pane rect installed, so clearing
  before disabling scissor would wipe only that pane and let the rest bleed.
  Then per pane: `paneViewport` → `gl.viewport`/`gl.scissor`, `ortho2d` from
  the pane's world `window`, dispatch each layer.
- **The dispatch seam (`Renderer.dispatchLayer`) is a `switch
  (descriptor.kind)` with NO arms yet.** Tasks 6–7 add `case
  "candle-bodies"` / `"candle-wicks"` / `"line-strip"` (and 10–13 the rest);
  each resolves its program via `getProgram(gl, key, factory)`, draws against
  the `projection`, and records the instance into `this.usedPrograms` so
  `dispose` releases it. The seam is in `dispatchLayer`, NOT in `drawPane`,
  so later tasks plug in without touching the per-pane loop. Today every kind
  is a graceful no-op.
- **The renderer is OPTIONAL — no `gl` ⇒ no `Renderer` ⇒ no-op draw.** The
  factory builds a `Renderer` only when a `GlContext` is available;
  `onEmissions` skips the draw entirely when there is none, so the headless
  default export + conformance + node tests still pass with a bare
  `{ width, height }` canvas. This is the headless-constructible invariant
  preserved end-to-end.
- **`gl` resolution + dispose ownership.** `resolveGlContext` precedence: the
  injected `opts.gl` test seam (wrapped in a structural `GlContext` via
  `wrapInjectedGl` — `resize`/`dispose` no-ops, dims from `opts.canvas` +
  `dpr`) FIRST; else a real `<canvas>` element (`getContext`) →
  `createGlContext` + `resize` (browser-only, `/* v8 ignore */`d); else a bare
  `{ width, height }` → `undefined`. A `created` flag tracks whether the
  factory built the context, so `dispose` tears down ONLY the
  factory-created one (the injected-seam context's lifetime belongs to the
  test that supplied it).
- **Single-overlay-pane MVP layout (SUPERSEDED by Task 10's subpane split —
  see that section).** Task 5 handed `buildFrame` one full-canvas overlay
  `PaneLayoutRect`; Task 10 replaced the factory's `layout()` with
  `computePaneLayout(state.paneOrder, …)`, which still collapses to a
  full-canvas overlay when there are no subpanes (so the MVP behaviour holds
  for a single-pane chart) but stacks subpanes below it otherwise. Each
  `PaneRenderState` now also carries an optional `cssRect` the Renderer scopes
  its viewport from.
- **`opts.palette` is now on `CreateWebglAdapterOpts`** and threaded into
  `createAdapterState` (conditional-spread for `exactOptionalPropertyTypes`),
  alongside `opts.initialVisibleBars`. `dispose` calls `resetAdapterState`
  (retains `view` + `palette` for a same-canvas remount).
- **The factory state is now the full Task-4 `AdapterState`** (was the
  Task-1 `{ bars }` stub). `runWebglLoop`'s `applyCandleEvent` mirrors candle
  events into `state.bars` unchanged; the `HANDLE_STATE` WeakMap now holds the
  real state.

## Candle programs (Task 6) — `src/webgl/programs/`

The first GPU program arms behind the Task-5 dispatch seam. Each `*-program.ts`
carries the 4-line invinite provenance header (SHA `cd883292`); `candle-pack.ts`
(+ its test) is a chartlang-native pure helper (2-line MIT header). The `gl.*`
program internals are browser-only (`/* v8 ignore */`, not coverage-gated); the
pure packers are node-unit-tested.

- **`base-program.ts` is the GENERAL lifecycle, kept candle-agnostic so Task 7's
  line-strip extends it.** `BaseProgram<T extends BaseDescriptor>` owns
  `Program` + `BufferPool` + `vaoCache` and runs `pack → acquire → upload
  (dirty-range-hint-aware) → buildVao (first per-pane miss) → setUniforms →
  onBeforeDraw? → bind → drawCall → unbind`. Optional hooks (`onBeforeDraw` /
  `drawOverride` / `prunePaneOverride` / `cleanupExtras`) let a divergent
  program replace pieces. `DrawArgs<T>` carries `{ descriptor, projection,
  viewportWidthPx, viewportHeightPx, dpr, paneKey }` — the `viewport*Px` feed the
  `project32` device-px snap, `dpr` scales the CSS-px widths. invinite's
  `import.meta.env.DEV` shared-pool-hash check + `tcLog` were dropped (Vite-only;
  see `buffer-pool.ts`). The dirty-range hint plumbing is kept for correctness
  but is dormant (our descriptors carry no `dirtyOffset`/`dirtyLength` yet ⇒
  every upload is a full `bufferData`).

- **Descriptor shape is the contract — both programs DIVERGE from invinite's:**
  - **Bodies:** our `rows = [x, open, high, low, close, isBull]` (6-float stride)
    matches invinite, so the port is faithful. The body width is the shared
    TradingView `computeBarWidthPx(barPitchPx, …)` formula
    (`src/webgl/lib/bar-width-formula.ts`, ported from invinite) — but we run it
    UPSTREAM in `buildFrame` (where the visible window + pane CSS width are known)
    rather than in-shader, so the descriptor's `bodyWidthPx` is already the
    pitch-resolved CSS-px width: bodies **shrink to avoid overlap when zoomed out**
    and cap at `CANDLE_BODY_MAX_WIDTH_PX = 6` (the ceiling fed as `maxWidthPx`,
    `wickClearancePx: 1`) when zoomed in. So the shader feeds `uBodyWidthPx`
    straight through and **floors the body at 1 device-px** (`max(1.0,
    uBodyWidthPx * uDpr)`, canvas2d `MIN_BODY_WIDTH_PX` parity) + inflates dojis
    to 1 device-px via `dojiInflateNdcY()`. 1 instance/bar.
  - **Wicks:** our `CandleWicksDescriptor` is a SINGLE quad per bar spanning
    `low→high` (`rows = [x, low, high, isBull]`, 4-float stride, `bullColor` +
    `bearColor`) — NOT invinite's 2-stub upper/lower model with a separate
    `bullFlags` Uint8 stream + `bodyBottom`/`bodyTop`. So the wicks program runs
    the CANONICAL `BaseProgram` flow (`drawArraysInstanced(…, rowCount)`, 1/bar);
    invinite's `drawOverride`/`prunePaneOverride`/bull-flags machinery is NOT
    ported. The edge-aligned X snap (left-edge to integer device-px, NOT a center
    snap) IS kept — a thin 1-CSS-px wick stays fully covered (crisp) at any DPR.
    The per-bar `isBull` flag is BOUND (`aIsBull`): the shader colours each wick
    via `mix(bear, bull, aIsBull)` so it matches its candle body's direction
    (`buildFrame` feeds the wick the SAME `candleBullBody`/`candleBearBody`
    palette colours as the bodies). The `palette.candleWick` token is now used
    only for axis-label text (`overlay.ts`), not the wick fill.

- **`program-cache.ts` gained `evictProgram(gl, key)`** — the surgical
  single-key evict (remove the stale instance WITHOUT disposing; the program's

  dispose-and-drop. Both candle programs are per-`gl` singletons resolved via

## Line-strip program (Task 7) — `src/webgl/programs/`

The plot-line GPU arm behind the Task-5 dispatch seam. `line-strip-program.ts`
and `line-strip-pack.ts` (+ its test) both carry the MIT header + the invinite
provenance comment (SHA `cd883292`): `line-strip-pack.ts` ports invinite's core
polyline packer and adds the chartlang-native `sampleMonotoneRuns` shared-spline
run sampler below the same header. The `gl.*` draw flow is browser-only
(`/* v8 ignore */`); the pure packer + sampler are node-unit-tested.

- **`line-strip-pack.ts` is pure: `packLineStrip` + `sampleMonotoneRuns`.**
  `packLineStrip` walks an N-point world polyline into the 12-float-stride
  per-segment instance buffer (`[prev.xy, current.xy, next.xy, further.xy,
  arclengthStart, arclengthEnd, pad, pad]`) + a per-point cumulative arclength
  that **carries the last finite value across a NaN gap** (a propagated NaN
  collapses `mod(vArclength, period)` and turns a dashed stroke solid); the raw
  NaN endpoints stay in `aCurrent`/`aNext` so the vertex shader's
  `nan_skip_segmentInvalid` collapses the segment to zero-area. invinite's
  horizontal/vertical-threshold packers were NOT ported (this adapter has a
  single `LineStripDescriptor`; hlines render via a later arm).
- **Default `line` smoothing reuses the SHARED spline — never fork it.**
  `sampleMonotoneRuns` densifies a `line` series through adapter-kit's
  `monotoneCubicSegments` (promoted from canvas2d in Task 7) into denser
  line-strip points (a few per gap), so the GPU draws the same smooth curve the
  other five adapters render by default. `buildFrame.buildLineStrip` samples
  only for `line` style; `step-line` (and area edges) stay STRAIGHT (no
  sampling). NaN gaps split runs (each finite run sampled independently); a
  <3-point run passes through verbatim; `xShift` is applied to the source points
  (via `shiftedBarTime`) BEFORE sampling. The y-scale contribution is unchanged
  (`computeYRange` reads the original series, not the densified points).
- **`line-strip-program.ts` extends `BaseProgram` via `drawOverride`** (a single
  shared VAO + a single shared `scratch` instance pool slot serve every pane —
  every line descriptor uploads its own bytes; no per-pane VAO cache). The
  vertex shader computes the miter direction from neighbour segment normals
  (`1 / max(0.1, dot(miter, normal))` scale; falls back to the segment normal
  when a neighbour is NaN/coincident via `nan-skip`), `vArclength` is `highp`
  for dash precision, the fragment dashes via `mod`. Per-`gl` singleton via
  `getProgram(gl, "line-strip", …)` + `evictProgram` (the candle pattern), NOT
  invinite's typed program-cache record.
- **`pxToWorldX/Y` are DERIVED, not passed in.** invinite passed
  `pxToWorldX/Y` through `DrawArgs`; our `DrawArgs` carry `projection` +
  `viewportWidthPx/HeightPx` + `dpr`, so `drawOverride` derives world-per-
  device-px from the column-major `ortho2d` matrix: `pxToWorldX = 2 /
  (uProj[0] * viewportWidthPx)`, `pxToWorldY = 2 / (uProj[4] *
  viewportHeightPx)`. The descriptor's `widthPx` / `dash` are CSS-px, so the
  shader's `uHalfWidthPx` = `widthPx * 0.5 * dpr` and the dash on/off are
  scaled by `dpr` (arclength is device-px).
- **The dispatch seam stays additive.** `Renderer.dispatchLayer`'s
  `case "line-strip"` resolves the program, `usedPrograms.add`, and `draw`s with
  the same extended `DrawArgs` the candle arms use — Tasks 8 / 10–14 plug their
  arms into the same `switch` without reworking this one.

## Filled-band program (Task 11) — `src/webgl/programs/`

The band-fill GPU arm behind the Task-5 dispatch seam — renders `filled-band`
overlays (Bollinger / `draw.fillBetween` ribbons) and the `area` plot fill body
as an alpha-blended triangle strip between two world-space edge polylines.
`filled-band-program.ts` carries the 4-line invinite provenance header (SHA
`cd883292`); `filled-band-pack.ts` (+ its test) is a chartlang-native pure
helper (2-line MIT header). The `gl.*` draw flow is browser-only
(`/* v8 ignore */`); the pure packer is node-unit-tested.

- **`filled-band-pack.ts`'s `packFilledBand(descriptor)` is pure: two world
  edge polylines → a NaN-gap-split triangle-strip vertex buffer.** It walks the
  descriptor's `upper` / `lower` (`[x, y, …]` each), and for every column where
  BOTH edges are finite appends two `[x, y]` vertices (upper then lower — the
  TRIANGLE_STRIP convention). A non-finite upper OR lower y closes the current
  run (no triangle spans the gap); the next finite column opens a fresh run.
  Returns `{ vertices, runs: { vertexOffset, vertexCount }[] }`. Mismatched edge
  lengths clamp to the SHORTER; empty / all-gap ⇒ zero vertices + no runs.
- **The descriptor DIVERGES from invinite — single uniform color, world x baked
  in.** invinite's `FilledBandDescriptor` carried bar-index `firstIndex` +
  `fillColorAbove` / `fillColorBelow` hex strings and packed per-vertex color so
  a run crossing `upper < lower` could swap palette mid-strip (Ichimoku bearish
  cloud). OUR descriptor (Task 4) carries world-space `upper` / `lower` arrays +
  ONE `color: RgbaUnit`, so the packer packs `[x, y]` vec2 vertices only and
  color rides a `uColor` uniform. The above/below color swap is dropped
  ("translate, not transcribe").
- **`filled-band-program.ts` extends `BaseProgram` via `drawOverride`** (the
  line-strip pattern): pack → acquire a per-descriptor pool slot → upload →
  build a vec2 `aWorldPos` VAO (first per-slot miss) → set `uProj` + `uColor` →
  alpha blend → one `gl.drawArrays(TRIANGLE_STRIP, offset, count)` per run.
  World pos projects straight through `uProj` with NO sub-pixel snap (band edges
  must not shimmer under pan/zoom — invinite's contract). Per-`gl` singleton via
  `getProgram(gl, "filled-band", …)` + `evictProgram`; `pack` / `buildVao` /
  `setUniforms` throw (drawOverride owns the flow).
- **`buildFrame` builds the band + area fill descriptors in the `(z, seq)`
  pass.** `buildFilledBand` packs a `filled-band` series' per-bar `upper` /
  `lower` (each `null` ⇒ a per-column NaN gap) with `style.alpha` on the color.
  `buildAreaFill` packs an `area` series' FILL body — `upper = value`,
  `lower = floorY` (the pane floor = bottom of the visible window, canvas2d
  parity) with `style.fillAlpha` — AND its existing `line-strip` edge (fill
  pushed first so the edge draws on top; both share the series' z/seq, and the
  sort is stable). `buildPaneLayers` now takes the resolved `floorY`. A
  `value:null` / `colorValue:null` gap packs NaN edges the GPU run-splitter
  skips.

## Drawings via decomposeDrawing → overlay (Task 13) — `src/`

The `draw.*` surface (all 63 kinds). `drawings.ts` (+ its test) is
chartlang-native (2-line MIT header) — no invinite port (the geometry IS
the shared `decomposeDrawing`). Pure mapping is node-tested; the 2D paint is
browser-only.

- **Drawings paint on the 2D text overlay through the SHARED
  `decomposeDrawing` + `paintPrimitive` — NOT a GL program; NO per-kind
  geometry is re-derived (the adapter-kit contract).** `drawings.ts`'s
  `drawingPrimitives(state, info)` is PURE: it builds the overlay-pane pixel
  `Viewport` via a `drawingViewport(state, info)` wrapper over `glyphs.ts`'s
  `paneViewportFromInfo(info)` — iterates `state.drawings`
  ordered by `(z ?? 0, drawingSeq)`, and concatenates
  `decomposeDrawing(drawing, viewport)` per live drawing into a flat
  `DrawPrimitive[]`. The factory's `onAxes` hook (overlay pane) paints them
  via `paintOverlayDrawings` → `overlay.paintDrawings(prims)` →
  `paintPrimitive(ctx, prim)` — byte-consistent with canvas2d / uplot / lwc.
- **`drawingViewport` re-bases the x window from bar SLOT to bar TIME — the
  ONE place the overlay diverges from the GL/glyph slot space.** The GL
  geometry, glyph anchors (`glyphs.ts`), and override bands (`overrides.ts`)
  all use the COMPRESSED bar SLOT (index) as world x (`barX` = `barIndex +
  xShift`; no calendar gaps), and project the slot directly via `timeToX(slot,
  …)`. But the SHARED `decomposeDrawing` projects each drawing's bar-TIME
  anchors (it is renderer-agnostic). So `drawingViewport` rewrites only the
  slot-space viewport's `xMin`/`xMax` to the bar TIME at those slot edges via
  `slotAnchorTime` (the inverse of the bar→slot mapping: interior slots
  interpolate between adjacent bar times, out-of-range slots extrapolate with
  the edge spacing). A drawing anchored at a bar's time then lands on the SAME
  pixel as that bar's candle, for ANY spacing. `y`/`pxWidth`/`pxHeight` (the
  slot-space projection) are unchanged; `< 2` bars returns the slot viewport
  unchanged (no time basis). WITHOUT this re-base a ~1e12 epoch anchor projects
  against the ~`[len−N, len−1]` slot window and flies ~1e12 px off-canvas (the
  "webgl drawings invisible" bug). Glyphs / overrides need NO change — they are
  already slot-native.
- **NO GL arc tessellation / no `drawings-program.ts` — the overlay sink
  paints arcs / fills / text / markers natively.** `decomposeDrawing` emits
  pixel-space primitives INCLUDING real `kind:"arc"` (circle / time-cycles /
  trend-angle / fib arcs), so routing them to the world-space GL line-strip
  program would need a pixel→world inverse (fighting the "ONE projection
  source" rule) AND arc tessellation. Painting on the overlay (whose
  `RenderCtx.arc` is native) avoids both — the task's preferred "less
  surface" outcome. The optional `arcToPolyline` helper + GL arc program in
  the task's Files table are therefore NOT built.
- **`op:"remove"` is honoured at ingest** (`applyDrawing` deletes from
  `state.drawings` + `state.drawingSeq`), so `drawingPrimitives` only ever
  iterates live drawings. Overlay pane only (the glyph precedent; the
  overlay pane is `paneOrder` index 0 with CSS origin `(0,0)`, so the
  origin-relative pixel coords need no `cssRect` offset). Independent of
  `state.bars` — a drawing can anchor by absolute time with no bars in the
  window (so the paint gate is `state.drawings.size > 0`, not bar count).
- **The `drawing` dispatch arm IS armed in `Renderer.dispatchLayer`
  (additive) but `buildFrame` builds NO `DrawingDescriptor` — drawings paint
  once, on the overlay.** The arm resolves the line-strip program for each
  `DrawingDescriptor.strokes[i]` (a world-space `LineStripDescriptor`) a
  caller supplies explicitly — the OPTIONAL world-space GL stroke path,
  mirroring the `marker` arm (armed, never auto-built, no double-paint).
  Since `drawing` is now armed, the Renderer-unit-test's un-armed-kind
  sentinel moved from `drawing` to `text` (the last un-armed kind — text is
  overlay-only, never dispatched through the GL pipeline). Task 14 owns the
  global z-order pass that interleaves drawings among the other kinds.

## Overrides + z-order pass + volume-profile (Task 14) — `src/`

The parity-completing layer: the candle / background overrides, the per-pane
z-order render pass, the deferred horizontal-volume-bars, and the final docs.
`renderOrder.ts` / `overrides.ts` (+ their tests) are chartlang-native (2-line
MIT header) — no invinite port (the comparator IS the shared adapter-kit
helper; the override semantics mirror canvas2d byte-for-concept). Pure mapping
is node-tested; the 2D paint is browser-only.

- **The z-order comparator is the SHARED `sortByRenderOrder` + `RENDER_BAND`
  from adapter-kit — NEVER a forked `a.z - b.z || a.band - b.band || a.seq -
  b.seq`.** `renderOrder.ts` exposes only the webgl mark payload + bands:
  `BAND` aliases `RENDER_BAND` (`{ series:0, glyph:1, hline:2, drawing:3 }`),
  `RenderOrderMark<T>` is `RenderOrderKey & { payload: T }`, and
  `applyRenderOrder(marks)` calls the shared comparator (stable, in place) and
  returns the payloads in order. Each WebGL draw SURFACE sorts its own marks
  with this one comparator: the GL pipeline's series + hline descriptors sort in
  `buildFrame.buildPaneLayers`; the 2D overlay's glyphs (`overlaySeq`) +
  drawings (`drawingSeq`) sort in their own passes. Default `z = 0` reproduces
  the canonical band order (series → glyph → hline → drawing); a `z` override
  reorders globally **within a pane** (never cross-pane). Substrate paints
  BEFORE the sorted pass; alert badges AFTER.

- **`buildPaneLayers` now renders horizontal lines too, and sorts via the
  shared comparator.** Each series mark is tagged `band: BAND.series`; each
  `hline` becomes a flat 2-point world-space `line-strip` (`[xMin, price, xMax,
  price]` across the pane window) tagged `band: BAND.hline` — so the canonical
  band order is REAL (series below hlines at the default z), not just a comment.
  A non-solid `lineStyle` dashes via the line-strip dash channel; the hline
  folds into the y-range upstream (`computeYRange`), so its price is always in
  window. `buildPaneLayers` takes the resolved `PaneWindow` (built before the
  layers) so the hlines span the visible x window. The local
  `entries.sort(...)` is gone — `applyRenderOrder` is the one sort site.

- **Overrides paint as SUBSTRATE on the 2D overlay, over the GL candles, BEFORE
  the sorted glyph / drawing pass** (the canvas2d `renderBackgroundOverlays` /
  `renderBarOverlays` order). `overrides.ts` is PURE: `resolveOverridePaint`
  reads the `bg-color` / `bar-color` / `bar-override` / `candle-override`
  emissions buffered in `state.plotOverlays` (Task 4 routed them there) and
  resolves them into CSS-px paint items projected through the overlay-pane
  `paneViewportFromInfo` (the SAME projection the glyphs / drawings / axis
  labels use — ONE source per frame). Semantics mirror canvas2d byte-for-concept
  (re-implemented locally; the no-cross-example-`src`-import rule forbids
  importing canvas2d's):
  - `bg-color`: a translucent full-pane-height band per bar (`alpha = 1 -
    transp/100`); the per-bar `colorValue` wins over `style.color`, a `null`
    is the explicit no-fill gap.
  - `candle-override`: bull / bear / doji by bar DIRECTION (`close > open` /
    `close < open` / else `doji ?? bull`) — NOT a whole-series tint.
  - `bar-color` / `bar-override`: a stroked OHLC outline; `bar-color` prefers
    `colorValue` over `style.color` (`null` gap), `bar-override` carries only
    `style.color`. Overrides anchor on the bar's OWN time (candle state, not a
    shifted series — `xShift` never applies); an override whose bar is absent
    from `state.bars` is skipped. `overlay.ts` gained `paintBackgroundOverlays`
    / `paintBarOverlays` (the low-level CSS-px paint); the factory's `onAxes`
    (overlay pane) calls `paintOverlaySubstrate` before the glyph / drawing
    passes.

- **Horizontal-volume-bars (the Task-10 deferral) RESOLVED on the 2D overlay,
  NOT a new GL program.** `resolveHorizontalHistogram` projects each
  `horizontal-histogram` bucket to a right-edge row (`width = volume / maxVolume
  * HORIZONTAL_HISTOGRAM_MAX_WIDTH_PX (96)`, `rowHeight 6`, grows leftward from
  the plot's right edge, palette `plotDefault` fallback) — canvas2d's
  `drawHorizontalHistogram` byte-for-concept. This is the canvas2d-consistent
  "less surface" outcome the task file prefers (matches the Task-13
  drawings-on-overlay precedent): a GL program + a 4-color value-area
  `HorizontalVolumeBarsDescriptor` was the alternative and is NOT built.
  `overlay.ts` gained `paintHorizontalHistogram`; the factory paints it in
  `paintOverlaySubstrate`. `isGlyphOverlay` still EXCLUDES h-histogram +
  overrides (they paint in the substrate pass, not the glyph band) — divergence
  from canvas2d, which folds h-histogram into its sorted glyph band.
