# Task 7 ÔÇö Coordinate resolver: `bar_index` Ôćö `time` + future-bar synthesis

> **Status: TODO**

## Goal

Bridge Pine's bar-index coordinate system (and `chart.point` factory
calls) to chartlang's absolute `(time, price)` `WorldPoint`s.
Historical bar_index references resolve via the runtime bar stream
(`bar.time` lookup with offset); future bar_index references (Pine
permits `bar_index + N` up to N=500) synthesize a `time` via
`bar.time + N * barInterval`. `xloc.bar_time` references pass through
unchanged. This task ships the IR-level rewrite logic ÔÇö the codegen
in Task 16 emits the helper calls this task defines.

## Prerequisites

Task 5 (semantic analysis with `referencesBarIndex` /
`referencesFutureBarIndex` flags), Task 6 (mapping tables).

## Current Behavior

Pine `chart.point` and bar_index expressions exist as raw AST nodes
with no rewrite. No `WorldPoint` synthesis logic exists.

## Desired Behavior

A package-internal `resolveCoordinates(annotated: AnnotatedScript,
opts: ConvertOpts, diagnostics: DiagnosticCollector):
CoordinateResolution` API in `src/transform/coordinates.ts` returns a
`ReadonlyMap<ExpressionNode, ResolvedAnchor>` mapping every
coordinate-bearing expression (the args to `line.new`,
`label.new`, `box.new`, `polyline.new`, all `chart.point.*` factory
calls) to a `ResolvedAnchor` describing how Task 16's codegen should
emit the chartlang `WorldPoint`.

The resolver does NOT rewrite the AST ÔÇö it produces a side-table.
This keeps the AST immutable and makes the resolver's output
inspectable by tests and by Task 17's diagnostic framework.

## Requirements

### 1. The `ResolvedAnchor` IR

```ts
export type ResolvedAnchor =
    | { kind: "literal-world-point"; time: number; price: number }
    | { kind: "expr-world-point"; timeExpr: string; priceExpr: string }
    | { kind: "bar-index-historical"; offsetExpr: string; priceExpr: string }
    | { kind: "bar-index-future"; offsetExpr: string; priceExpr: string; requiresBarInterval: true }
    | { kind: "bar-time-direct"; timeExpr: string; priceExpr: string }
    | { kind: "chart-point-now"; priceExpr: string }
    | { kind: "chart-point-from-index"; offsetExpr: string; priceExpr: string }
    | { kind: "chart-point-from-time"; timeExpr: string; priceExpr: string }
    | { kind: "chart-point-new"; timeExpr: string; offsetExpr: string; priceExpr: string };
```

`*Expr` fields are **strings of chartlang TypeScript** ÔÇö the codegen
emits them verbatim into the output. Why strings? The `ExpressionNode`
union belongs to the Pine side; the converter's IR has to bridge to
chartlang TS source. Synthesizing strings here keeps codegen (Task 16)
purely templating and avoids a second expression AST. The strings are
generated by a small "pine-expr-to-ts-expr" mini-emitter that this
task ships (see ┬ž3).

### 2. Resolution rules

For each call-site that takes coordinate arguments, decide the kind
based on the source expression shape:

- Both `x` and `y` are compile-time literals Ôćĺ `literal-world-point`.
  Compute the literal `time` and `price` at convert time.
- The drawing's `xloc` arg is `xloc.bar_time` Ôćĺ
  `bar-time-direct`. Pine's `x` is already ms; pass through with
  `timeExpr` as the chartlang-emitted expression for `x`.
- The drawing's `xloc` is `xloc.bar_index` (default) and `x` is:
  - `bar_index` Ôćĺ `bar-index-historical` with offset 0.
  - `bar_index[N]` for literal `N > 0` Ôćĺ
    `bar-index-historical` with offset `N`.
  - `bar_index + N` for literal `N > 0` Ôćĺ
    `bar-index-future` with offset `N`. Sets
    `requiresBarInterval: true`.
  - `bar_index - N` for literal `N > 0` Ôćĺ
    `bar-index-historical` with offset `N`.
  - `bar_index` minus or plus a non-literal expression Ôćĺ
    `bar-index-future` or `bar-index-historical` depending on the sign
    inference; emit `dynamic-bar-index` warning.
  - Any other expression Ôćĺ emit `unresolved-bar-index` warning,
    fallback to `bar-index-historical` with `offsetExpr: "0"` (so
    output still compiles; user sees the warning).
- A `chart.point.*` factory call Ôćĺ one of the four
  `chart-point-*` kinds based on the chain match.

### 3. Pine-expr to chartlang-TS-expr emitter (`src/transform/exprEmit.ts`)

A small recursive walker that converts a Pine `ExpressionNode` to a
chartlang TS expression string. Handles:

- Literals (passthrough, with color literals re-emitted as quoted
  strings).
- Identifiers (passthrough; but `close` Ôćĺ `bar.close`, `volume` Ôćĺ
  `bar.volume`, etc. ÔÇö the OHLCV identifiers are remapped here using a
  small `BUILTIN_IDENTIFIER_MAP` from `src/mapping/builtinIdentifiers.ts`).
- Binary / unary / ternary operators (passthrough; `and`/`or`/`not` Ôćĺ
  `&&`/`||`/`!`).
- `chart.point.*` calls ÔÇö extract args.
- Function calls ÔÇö leave the function-name resolution to Tasks 9, 13,
  14, 15; here we only emit the textual call structure when called
  recursively for an argument expression.
- History access `x[n]` Ôćĺ `x[n]` (chartlang series indexing has the
  same shape).
- Member access ÔÇö passthrough.

The output is a string. Operator precedence is preserved by always
wrapping non-trivial sub-expressions in parens (cheap and safe; the
chartlang typechecker doesn't care).

### 4. `BUILTIN_IDENTIFIER_MAP` (`src/mapping/builtinIdentifiers.ts`)

```ts
export const BUILTIN_IDENTIFIER_MAP: ReadonlyMap<string, string>;
```

Entries:
- `open` Ôćĺ `bar.open`, `high` Ôćĺ `bar.high`, `low` Ôćĺ `bar.low`,
  `close` Ôćĺ `bar.close`, `volume` Ôćĺ `bar.volume`,
  `hl2` Ôćĺ `bar.hl2`, `hlc3` Ôćĺ `bar.hlc3`, `ohlc4` Ôćĺ `bar.ohlc4`,
  `time` Ôćĺ `bar.time`, `bar_index` Ôćĺ `__bar_index()` (a converter-
  emitted helper, see ┬ž6),
- `na` Ôćĺ `Number.NaN` is the **default emission** for numeric
  contexts. `exprEmit` MUST consult Task 5's `SemanticAnnotation.naKind`
  when emitting a bare `na` identifier: when the annotation is
  `"handle"`, emit `null` instead (chartlang's `DrawingHandle | null`
  sentinel). When the annotation is `"numeric"` or absent, emit
  `Number.NaN`. The map only encodes the default; the emitter applies
  the per-site override.
- `barstate.isfirst` Ôćĺ `barstate.isfirst`, etc. (passthrough ÔÇö same
  names),
- `xloc.bar_index` Ôćĺ `"bar-index"` (string sentinel for the resolver's
  use; never reaches output unless intentionally),
- `xloc.bar_time` Ôćĺ `"bar-time"` (same).

### 5. Future-bar synthesis

When the resolved kind is `bar-index-future`, the codegen emits:

```ts
{ time: bar.time + ((N) * __BAR_INTERVAL_MS), price: <priceExpr> }
```

`__BAR_INTERVAL_MS` is a converter-emitted constant that Task 16 wires
to a callsite-extracted `barInterval` value:

- If `ConvertOpts.barInterval` is non-null Ôćĺ emit the literal in the
  output script's top-level scope.
- If `ConvertOpts.barInterval` is null but the script references the
  future arithmetic Ôćĺ emit `requires-bar-interval` error diagnostic
  at convert time. The output script does not compile (deliberately
  fail-loud) unless the caller supplies the value.

A future enhancement (logged in README ┬ž9): infer `barInterval` from
`timeframe.in_seconds * 1000` when the script reads it; out of scope
for v1.

### 6. `__bar_index()` helper emission

The chartlang runtime does not expose a `bar_index` natively (only
`bar.time`). To support Pine scripts that reference `bar_index`
arithmetically (e.g. as the x-coord of a `line.new`), Task 16 emits a
helper at the top of the output script:

```ts
let __barCount = 0;
const __bar_index = (): number => __barCount;
```

ÔÇŽand increments `__barCount` at the top of `compute({...})`, **gated
on `barstate.isnew`** so the counter advances exactly once per bar
even when the runtime replays history (`onHistory`) before live
ticks. Without the gate, intra-bar ticks would double-count. Module-
level `let` persists across `compute(...)` calls within a mount
(confirmed by `drawInteractiveUpdate.scenario.ts`), so `__barCount`
accumulates monotonically per mount. Task 16 ships the helper;
Task 7 produces the string `"__bar_index()"` as the resolved
expression whenever a Pine `bar_index` reference appears.

Emit shape (Task 16 templating):

```ts
let __barCount = 0;
const __bar_index = (): number => __barCount;

// ...in compute:
if (barstate.isnew) __barCount++;
```

### 7. Diagnostic codes (added this task)

- `requires-bar-interval` (error) ÔÇö script uses `bar_index + N` future
  arithmetic but `ConvertOpts.barInterval` is null.
- `dynamic-bar-index` (warning) ÔÇö non-literal arithmetic on
  `bar_index`; offset interpretation is best-effort.
- `unresolved-bar-index` (warning) ÔÇö coordinate argument is not a
  recognized pattern; fallback applied.
- `chart-point-from-index-without-xloc` (warning) ÔÇö `chart.point.from_index`
  used with a drawing whose `xloc` is `bar_time` (Pine renders this at
  undefined position; the converter emits a warning + uses the index
  as-if `bar_index` mode).
