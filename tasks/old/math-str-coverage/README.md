# math.* / str.* coverage completion

## Overview

The `math.*` and `str.*` namespaces shipped in core (`@since 1.4`):
`packages/core/src/math/index.ts` (9 functions) and
`packages/core/src/str/index.ts` (14 functions). Their **language** and
**primitives** doc pages, the headline examples
(`tick-snapped-levels`, `str-formatted-hud`), the hover registry, and
the converter's `math.*` passthrough all landed with that work.

Three coverage gaps remain. This task folder closes them:

1. **Skills** — the generated agent reference
   `skills/chartlang-coding/references/primitives.md` has no `math.*` /
   `str.*` sections, because the generator
   (`scripts/generate-skills-reference.ts`) only walks `ta` / `draw` /
   plot-family sources. `SKILL.md`'s surface description is stale. The
   `skills:gate` passes silently because the committed file matches the
   incomplete generator — a documented root-`CLAUDE.md` invariant is
   violated.
2. **Docs navigation** — `docs/primitives/math.md` and
   `docs/primitives/str.md` generate correctly but are absent from the
   primitives sidebar in `docs/.vitepress/config.ts`, so they are
   unreachable from that nav.
3. **Pine converter** — `str.*` lowering (`packages/pine-converter/
   src/transform/strFormat.ts`) is an explicit "v1 subset" of 8 Pine
   functions. The full Pine v6 `str.*` surface is 18 functions; ten are
   unmapped (they warn + pass through). The `math.*` passthrough is
   already complete and is **out of scope**.

Plus an additive ask: **2 author examples** exercising the
math/str functions that no example currently demonstrates.

Per root `CLAUDE.md`: a behavior change must update the same folder's
`CLAUDE.md`, and changing anything a `skills/` file describes must
update that skill in the same PR.

## Architecture Decisions

| Decision | Rationale |
|----------|-----------|
| **Skills `math.*`/`str.*` render as one consolidated block per namespace** (not per-member like `ta.*`) | Mirrors the single-page `docs/primitives/math.md` / `str.md` shape; 9 + 14 small frozen members don't warrant per-member blocks. Reuses `parsePhase4DocEntry` (already exported from `genPhase4Docs.ts`). |
| **Skills generator reuses `parsePhase4DocEntry`, does not fork a parser** | The namespace-member JSDoc walk (`childrenFor`/`findSymbolTrace`/`resolveShorthand`) already exists and handles `Object.freeze({...})` shorthand members. Re-deriving it would duplicate logic the docs gate already guards. |
| **Converter emits native JS for new `str.*` mappings** (user-confirmed) | Matches the established convention (`str.upper`→`.toUpperCase()`, bare `Math.*` passthrough). No new `str` import/destructure; no divergence from the native-where-native-exists pattern. |
| **`str.tonumber(s)` → `Number(s)`; `str.match` / `str.format_time` reject** (user-confirmed) | `Number(s)` yields `NaN` ≈ Pine `na` on bad input (minor edge: `""`→`0`). `match` is regex (core deliberately avoids `RegExp` — ReDoS/determinism); `format_time` needs host time. Both reuse the existing `str-not-mapped` warn + passthrough. |
| **New examples are compile-only + demo-mirrored, no new conformance scenario** (user-confirmed scope: "to display the example") | Namespace byte-stability is already proven by the headline examples' scenarios (`math-round-to-mintick`, `strFormattedTable`). New examples are for discoverability; the CLI e2e compile gate + `examples:sync` are sufficient. |
| **2 comprehensive examples, not 4 or 1** (user-confirmed) | One math + one str maximizes per-namespace function coverage with minimal new gate surface (each new example adds an e2e path, a demo mirror, and a generated docs page). |
