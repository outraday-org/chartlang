### 2. Series-and-indexing guide (`docs/language/series-and-indexing.md`)

Add a "User-created series — `state.series`" section after the bar/`ta.*`
material:

- `state.series(init)` is a writable, indexable number series. Write the
  current bar's value with `s.value = expr`; read history with `s[1]`,
  current with `s.current` / `+s` / `s[0]`; `s.length` is the filled count.
- It advances once per bar automatically — `s[1]` is always one bar ago; a bar
  with no write is a `NaN` gap (carry forward with `s.value = +s`).
- Footgun note (mirror the bar-series one): `Number.isFinite(s)` is `false`
  and `s === 5` is `false` (object vs number) — use `s.current` / `+s` /
  `s.value`.
- **"Which do I reach for?" disambiguation** (this is the section that keeps
  the feature from looking redundant with directly-indexable-bar-series): to
  index a built-in or a `ta.*` output, index it **directly** (`bar.close[3]`,
  `ta.ema(bar.close, 14)[1]`) — you do **not** need `state.series`. Reach for
  `state.series` only when you need the history of a value **you** compute that
  is **not** already a series — especially a **self-referential recurrence**
  (a value defined in terms of its own prior bar) or a **conditionally-updated**
  value. Those genuinely cannot be expressed as `bar.close[N]`.
- A compiling code fence (checked by `pnpm docs:snippets`) — chosen to be
  self-referential so it is NOT expressible via `bar.*`:
  ```ts
  // Consecutive up-closes: the value depends on its OWN prior bar, so it
  // can't be read off bar.close[N]. state.series both stores it and lets
  // you look back at the streak N bars ago.
  const streak = state.series(0);
  streak.value = bar.close.current > bar.close[1] ? streak[1] + 1 : 0;
  plot(streak.current);
  ```

### 3. New example script + demo entry

- Create `examples/scripts/up-streak.chart.ts` (or similar) using
  `state.series` for a **self-referential** indicator that genuinely needs a
  writable series — i.e. a value defined in terms of its own prior bar, NOT
  derivable from `bar.close[N]`. This is the load-bearing choice that keeps the
  example from looking redundant with the `directly-indexable-bar-series`
  "Manual SMA" demo (which uses `bar.close[N]` directly). Concretely, a
  consecutive-up-close streak counter (`overlay: false`, a subpane), with a
  NaN-safe warmup guard:
  ```ts
  compute({ bar, state, plot }) {
      const streak = state.series(0);
      const up = bar.close.current > bar.close[1];
      streak.value = up ? (Number.isFinite(streak[1]) ? streak[1] : 0) + 1 : 0;
      plot(streak.current, { title: "Up streak" });
      plot(streak[3], { title: "Streak 3 bars ago" });
  }
  ```
  A "bars-since-event" counter (`since.value = signal ? 0 : since[1] + 1`) is
  an equally good alternative — pick whichever reads cleaner. Follow
  `examples/scripts/CLAUDE.md` conventions. Note this is deliberately a
  DIFFERENT script from Task 4's conformance scenario (which uses the
  `s.value = bar.close` equivalence as a checkable correctness proof) — the
  demo's job is to show a case that genuinely needs the feature.
- Add a `DEMO_SCRIPTS` entry in `apps/site/src/components/demo/scripts.ts`
  (`id: "up-streak"` or `state-series`, a `label`, `source`, and a catalogue
  `description` that states the disambiguation: "history of a value you compute
  yourself — here a self-referential streak — which `bar.close[N]` can't
  express"). Keep the `id` stable for the `?script=…` deep link.
- Run `pnpm examples:generate`, verify `pnpm examples:gate` (rewrites
  `docs/examples/<new>.md` + `index.md`; never hand-edit generated files).
