# `draw.fillBetween` — native fill-between-series primitive

## Overview

Add a first-class `draw.fillBetween(edgeA, edgeB, opts?)` drawing
primitive to chartlang: a filled ribbon between two edges (each a
sequence of `WorldPoint`s). This is the native band/fill primitive that
TradingView Pine Script expresses as `linefill.new(line1, line2, color)`
(fill between two line drawings) and `fill(plot1, plot2, color)` (fill
between two series).

Today chartlang ships **no** fill-between primitive. The pine-converter
lowers Pine's `linefill.new` best-effort onto `draw.rotatedRectangle`
(a box, not a true fill) and emits an approximation note; the dynamic
forms hard-reject (`linefill-over-ring`, `cross-collection-linefill`).
After this work the converter emits the real primitive, the docs /
examples / demo / skills surface it, and the draw `§22.10` set (JSDoc,
the unit test layer the sibling draw primitives carry, conformance
scenario, auto-generated docs page) lands with it.

References: root `CLAUDE.md` (skills mirror rule), `CONTRIBUTING.md`
§16.3 (test layers) + §22.10 (new-primitive landing contract),
`packages/core/src/draw/`, `packages/runtime/src/emit/draw/`,
`apps/CLAUDE.md` (DEMO_SCRIPTS → examples:generate), `scripts/CLAUDE.md`
(skills:generate).

## Target State

- `draw.fillBetween(edgeA, edgeB, opts?)` callable from author scripts;
  returns a `DrawingHandle` (so `.update(...)` re-anchors it each bar and
  `.remove()` deletes it — chartlang's tracking model, the same way the
  Camp A "tracking line" example re-anchors per bar).
- New `DrawingKind` member `"fill-between"`, camelCase `fillBetween`,
  bucket `"polylines"`, `FillBetweenState`, `FillBetweenStyle`.
- Canvas2d reference adapter renders it as a filled closed polygon
  (`edgeA` forward then `edgeB` reversed), stroke optional.
- Full draw §22.10 set: JSDoc (`@anchors`, `@anchorCount`, `@bucket`,
  `@since`, `@stable`, `@example`), the test layers the **sibling draw
  primitives actually carry** (a co-located unit `*.test.ts` — `path`
  and `line` ship neither a `.property.test.ts` nor a `.bench.ts`; the

  scenario with pinned hash, and the auto-generated
  `docs/primitives/draw/fill-between.md` (kebab-case filename derived
  from the kebab kind — generated pages are kebab, e.g.
  `rotated-rectangle.md`). A property test is optional/recommended;
  100% coverage on the new file is the hard gate.
- Pine `linefill.new(lineA, lineB, color)` lowers to
  `draw.fillBetween([a1,a2], [b1,b2], { fill: color })` — the
  `rotatedRectangle` approximation note is removed; the dynamic rejects
  stay.
- Narrative docs, an example script, a demo entry, and the skills
  reference all surface the primitive. All touched `CLAUDE.md` files
  updated. Changeset committed.

## Architecture Decisions

| Decision | Rationale |
|----------|-----------|
| **Edges are `ReadonlyArray<WorldPoint>` snapshots, not references to two line handles** | chartlang drawings track motion by the script re-anchoring them each bar via `handle.update(...)` (the tracking-line idiom), NOT by the renderer resolving cross-drawing references. Snapshot edges keep `fill-between` self-contained — no new "resolve another drawing by id" capability in the adapter/wire — while producing the same visual as Pine's auto-tracking linefill. `handle.id` exposes no anchors to script code anyway, so a handle-pair API is impossible without a new read-back channel. |
| **Name `fillBetween` (kebab kind `fill-between`)** | User-selected. Unambiguous about the two-edge semantics; avoids colliding with the `fill` / `fillAlpha` style fields. |
| **Filled region = closed polygon `edgeA[0..n]` then `edgeB[m..0]`** | Robust for any two polylines (works for the 2+2 quad of a Pine linefill and for multi-vertex bands); no requirement that the two edges share x-coordinates or equal length. Matches how `draw.path` closes. |
| **Bucket `"polylines"`** | Two-polyline-shaped, consistent with `path` / `polyline` / channel kinds. Pine charges linefills no separate cap; `polylines` (Pine `max_polylines_count` 100) is the closest analogue. |
| **Variable anchor count (`@anchorCount` "variable")** | Mirrors `draw.path` / `draw.polyline`; the band may have 2..N vertices per edge. |
| **NaN / degenerate handling = skip, no throw** | An edge with `< 1` point or a `NaN` price is a silent no-op for that frame, matching every other drawing's warmup behaviour (no error, no partial polygon). |
| **Converter snapshots both lines' anchor expressions into the two edges** | The converter already holds each line's anchor expressions at the `linefill.new` site, so it inlines them into `edgeA` / `edgeB` and re-emits per the owning camp — no handle read-back needed. |

## Deferred / Follow-Up Work

- **Gradient overload** (Pine's `fill(p1, p2, top_value, bottom_value,
  top_color, bottom_color)` vertical gradient) is **not** in scope — a
  solid `fill` + `fillAlpha` only. Track as a follow-up.
- **`plot`-level fill between two `plot()` series** (Pine `fill(plot1,
  plot2)`) stays out of scope; `draw.fillBetween` covers the drawing
  case. A `plot.fillBetween` could follow.
- **"One linefill per line pair" replacement semantics** (Pine replaces a
  duplicate linefill on the same pair) is not enforced — each
  `draw.fillBetween` call is an independent drawing. Document, defer
  dedup.
