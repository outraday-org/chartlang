# `request.security` higher-timeframe expression form

## Overview

Add an **expression/callback overload** to `request.security` so a
`ta.*` (or other series) expression is evaluated **in the
higher-timeframe (HTF) context**, the way Pine's
`request.security(syminfo.tickerid, "1W", ta.ema(close, 20))` works.

```ts
// new form — EMA computed over WEEKLY bars, aligned no-lookahead to the chart
const weeklyTrend = request.security({ interval: "1W" }, (bar) => ta.ema(bar.close, 20));
plot(weeklyTrend, { color: "#ef5350", title: "Weekly EMA(20)" });
```

Today only the **data-only** form exists:
`request.security({ interval: "1W" })` returns a `SecurityBar` (OHLCV
series aligned to the *main* timeline). Because that aligned series is
clocked to the main bars, `ta.ema(weekly.close, 20)` averages 20 **main**
bars of a weekly-stepped series — so a "weekly EMA(20)" comes out nearly
identical to the daily EMA(20). The new form runs the EMA on the weekly
clock (20 *weekly* bars ≈ 140 days), then aligns the result down to the
main timeline — the smooth, lagged HTF trend authors expect.

## Current State

- **Core** (`packages/core/src/request/request.ts`): `request.security`
  has one signature `security(opts: RequestSecurityOpts): SecurityBar`.
  `RequestSecurityOpts = { interval: string }`. `SecurityBar` is the
  12-field OHLCV-series struct. `ComputeContext.request`
  (`packages/core/src/types.ts`) exposes the namespace. The primitive is
  registered `slot: true` in `packages/core/src/statefulPrimitives.ts`.
- **Compiler**: `extractRequestedIntervals.ts` reads the literal
  interval into `manifest.requestedIntervals` and emits
  `request-security-interval-not-literal` for non-literals.
  `callsiteIdInjection.ts` already injects slot ids into *every* nested
  call (so `ta.*` inside a callback would get a slot id).
  `extractMaxLookback.ts` computes the ring-buffer capacity.
  `program.ts` carries the ambient core type shim; `diagnostics.ts` owns
  the diagnostic-code union. No callback/expression handling anywhere.
- **Runtime**: `request/security.ts` builds the aligned `SecurityBar`
  via `alignHtfSeriesCache.ts` (`getOrAlign`) + `alignHtfSeriesToLtf.ts`
  (no-lookahead) over `streamBars.ts` ascending arrays. Secondary
  streams are real `StreamState`s, fed bar-by-bar via `streamKey` candle
  events (`createScriptRunner.ts` `pushSecondaryEvent` →
  `execution/secondaryStream.ts`). `ta.*` primitives read/write state on
  `ctx.stream.taSlots` and always target `ctx.stream`. The
  DepRunner/SiblingRunner pattern (`dep/DepRunner.ts`) already runs a
  separate compute body per bar by swapping `ctx.stream` + the active
  context — the template for the new HTF runner.
- **Hosts**: `host-worker` (`buildBundleFromModule`) and `host-quickjs`
  (`dispatcherCore`) adopt the compiler `__manifest` sidecar for
  single-script modules.
- **Conformance**: `mtfRequestSecurityClose.scenario.ts` (happy path,
  `multiTimeframe: true`, `secondaryCandles: { "1D": … }`, plot-hash +
  diagnostic-absent) and `requestSecurityNanFallback.scenario.ts`
  (`multiTimeframe: false`, all-NaN + `multi-timeframe-not-supported`).
- **Examples/demo**: `examples/scripts/htf-trend-filter.chart.ts`
  (data-only form, `ta.ema(weekly.close, 10)`), its runtime test in
  `examples/canvas2d-adapter/src/integration.test.ts`, and the inlined
  demo copy `HTF_TREND_FILTER` in
  `apps/site/src/components/demo/scripts.ts`. The demo's `secondaryStreams.ts`
  resampler + `ChartPane.tsx` multi-stream pump already feed the weekly
  stream from the daily `bars.json`.
- **Docs**: `docs/language/multi-timeframe.md` (hand-written),
  `docs/examples/htf-trend-filter.md` (auto-gen from DEMO_SCRIPTS),
  `docs/converter/diagnostics.md` (auto-gen).
- **Skills**: `skills/chartlang-coding/SKILL.md` lists the 3-arg Pine
  form as **forbidden**. `references/primitives.md` is regenerated by
  `pnpm skills:generate`.
- **Pine converter**: `src/transform/requestSecurity.ts` lowers Pine's
  `request.security(sym, "D", ta.ema(close, 9))` to the **wrong**
  main-timeframe form `request.security({ interval: "1d" }).ta.ema(bar.close, 9)`.

## Target State

- `request.security` gains a **second overload**:
  `security<T extends number>(opts, expr: (bar: SecurityBar) => Series<number> | number): Series<number>`.
  The callback receives the HTF `SecurityBar` (raw HTF OHLCV series, on
  its own clock) and uses the **ambient** `ta`/`math` captured from the
  enclosing `compute` scope. The data-only overload is unchanged.
- The callback may reference **only** the HTF `bar` param, `ta`/`math`,
  `inputs`, and literal constants. Capturing any other outer binding is
  a new compiler error `request-security-expr-captures-local`.
- The compiler hoists each expression callsite into a registerable
  **security-expression unit** keyed by `slotId|interval`, recorded in
  the manifest. `extractMaxLookback` accounts for lookback inside the
  callback so the secondary stream is sized correctly.
- The runtime drives a **`SecurityExprRunner`** (DepRunner-style) **on
  each HTF bar close**, against the secondary `StreamState`, producing
  one output value per HTF bar into an output buffer. `request.security`
  returns that output series aligned no-lookahead to the main timeline.
- Both hosts boot the hoisted unit. Conformance proves the weekly value
  differs from a same-length main-timeframe EMA. The example, demo,
  docs, skills, and Pine converter all move to the callback form.

## Architecture Decisions

| Decision | Rationale |
|----------|-----------|
| **Callback form `(bar) => …`, not "series in a variable"** | A `SecurityBar` series is clocked to the *main* timeline; any `ta.*` applied to it counts its window in main bars, not HTF bars (the root bug). Running the expression on the HTF clock is the only correct way; matches Pine. |
| **Ambient `ta`/`math`, HTF `bar` param** | Closest to Pine ergonomics. The runtime swaps `ctx.stream` to the secondary stream while the callback runs, so the ambient `ta` routes to the HTF stream automatically. |
| **Restricted scope (no outer-local capture)** | Keeps the hoist deterministic and side-effect-free. Capturing a main-timeline local would smuggle the wrong clock back in. Enforced by a compiler diagnostic. |
| **Drive on HTF bar close (not main close)** | `ta.*` are incremental/stateful (one append per bar). Driving once per HTF close makes `ta.ema(bar.close, 20)` accumulate over weekly bars. |
| **Reuse `getOrAlign` / `alignHtfSeriesToLtf`** | The output series is just another HTF series to align no-lookahead to the main timeline — identical to OHLCV alignment today. |
| **`SecurityExprRunner` modeled on `DepRunner`** | The sub-runner pattern (own RuntimeContext, swapped `ctx.stream`, suppressed emissions) is exactly what an HTF expression needs. |
| **Return `Series<number>` (single numeric series) in v1** | Covers the dominant case (`ta.ema`, `ta.sma`, `ta.rsi`). Multi-value returns (e.g. `bb`) deferred. |

## Provenance

No `../invinite/` port. This is a chartlang-native language feature
modeled on Pine's `request.security(sym, tf, expr)` semantics.

## Deferred / Follow-Up Work

- **Multi-value expression returns** (e.g. `request.security(opts, (b) => ta.bb(b.close, 20))`
  returning a struct of series) — v1 returns a single `Series<number>`.
- **Cross-symbol requests** — still locked to the chart symbol.
- **`lookahead` option** — chartlang stays no-lookahead only.
- **`request.lowerTf` expression form** — out of scope; only
  `request.security` gains the callback.
- **HTF-aware secondary-stream capacity tightening** — v1 sizes the
  secondary stream from the whole-file `maxLookback` (conservative);
  a follow-up could size it from the callback's own lookback only.
