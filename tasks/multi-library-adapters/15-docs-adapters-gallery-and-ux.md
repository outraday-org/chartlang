# Docs adapters gallery (generated) + README standardization + UX polish

> **Status: TODO**

## Goal

Give users a single, always-accurate place to discover and choose an
adapter. Extend the Task-14 generator to emit a **generated**
`docs/adapters/gallery.md` from the adapters registry — a comparison
matrix, per-adapter cards with GitHub deep links + one-command install,
and a "which adapter should I choose?" guide — gated by `adapters:gate`.
Standardize the five adapter READMEs, and point the root README +
first-adapter guide at the gallery and the `add-adapter` installer.

## Prerequisites

- Task 13 (docs `/adapters/` index + per-library reference pages + nav).
- Task 14 (registry, `gen-adapters.ts`, `adapters:gate`, `add-adapter`).

## Current Behavior

After Task 14 the registry + generator produce the CLI bundle and a
terminal `--list`, but there is no docs gallery page; discovery is split
across `docs/adapters/writing-an-adapter.md`, the per-library pages, and
GitHub. No comparison matrix exists. Adapter READMEs were each authored
independently (Tasks 4/6/8/10/12).

## Desired Behavior

`docs/adapters/gallery.md` is auto-generated from the registry and linked
from the Adapters nav; it shows every adapter, its capabilities, license,
render tech, GitHub folder, and the exact install command. The five
adapter READMEs share one template. Hand-edits to the gallery fail the gate.

## Requirements

### 1. Gallery generator — extend `scripts/gen-adapters.ts`

Add a second output: `docs/adapters/gallery.md`, emitted from
`ADAPTERS` (the same SSOT used for the CLI bundle), with a
`<!-- GENERATED by pnpm adapters:generate — DO NOT EDIT -->` header. Page
structure:

- **Intro** — what an adapter is (1 paragraph) + the two ways to get one:
  `npx @invinite-org/chartlang-cli add-adapter <id>` (full library adapter)
  vs `scaffold-adapter` (blank starter). Link `writing-an-adapter` +
  `conformance`.
- **Comparison matrix** — one row per adapter:
  | Adapter | Library | License | Render tech | Drawing surface | ~Bundle | Source |
  with `Drawing surface` = "Full (62 + all plots)" and `Source` a GitHub
  deep link to `examples/<dir>/` (via `githubFolder(entry)`).
- **Per-adapter cards** — for each entry: display name, `bestFor`,
  integration strategy (ctx / nodes / graphic), install command
  (`npx … add-adapter <id>`), and three GitHub deep links: folder, README,
  and the committed conformance report. (Render `approxBundleKb`, license,
  render tech from the registry.)
- **"Which should I choose?" guide** — a short decision list keyed off
  `bestFor` / `strategy` (financial-native → lightweight-charts;
  tiny/fast → uplot; dashboards/ecosystem → echarts; interactive/custom →
  konva; zero-dep reference → canvas2d).
- **FAQ / troubleshooting** — examples are private (copy/`add-adapter`, not
  `npm install`); the linear-price-axis caveat for non-canvas adapters;
  where the geometry layer lives (`@invinite-org/chartlang-adapter-kit`).

### 2. Extend the gate — `pnpm adapters:gate`

The gate (from Task 14) now byte-diffs **both** the CLI bundle and
`docs/adapters/gallery.md`. Add `gallery.md` to the generator's diff set.

### 3. VitePress wiring — `docs/.vitepress/config.ts`

Add `{ text: "Gallery", link: "/adapters/gallery" }` to the `/adapters/`
sidebar (top of the Adapters group) and point the navbar "Adapters" entry
at `/adapters/gallery` (the discovery landing). Update `docs/CLAUDE.md`:
list `docs/adapters/gallery.md` among the **auto-generated** pages
(hand-edits fail `adapters:gate`), alongside `primitives/ta` and `examples`.

### 4. Standardize the five adapter READMEs

Apply one template to `examples/{canvas2d,lightweight-charts,uplot,echarts,konva}-adapter/README.md`
(each ≤ 100 lines): title + one-line purpose; a badge row (license,
"copy-only — not published", render tech, "full conformance"); **Get it**
(`npx … add-adapter <id>` + the GitHub folder link); **Public surface**
(`createXAdapter`, `X_CAPABILITIES`, default export); **How drawings
render** (one line: shared `decomposeDrawing` → ctx/nodes/graphic);
**Docs** (link to the gallery + the per-library page at
`/adapters/reference/<id>`); license. Keep it
hand-authored (library-specific), but consistent — the generated gallery
is the SSOT for cross-adapter comparison, so the READMEs need not be generated.

### 5. Root README + first-adapter guide

- Root `README.md` (≤ 300 lines): an "Adapters" section with the
  one-command install line and a link to the gallery (supersedes the bare
  list added in Task 13).
- `docs/getting-started/write-your-first-adapter.md`: add `add-adapter
  <library>` as the fast path (full adapter) next to `scaffold-adapter`
  (blank starter); link the gallery.

### Edge cases

- `approxBundleKb` is a hand-maintained registry field (not auto-measured)
  so the gate stays deterministic — document this in the gallery FAQ and
  the registry.
- A new adapter added to the registry must appear in the gallery after
  `adapters:generate`; `adapters:gate` is red until regenerated.
- GitHub deep links must use `main` + the canonical
  `github.com/outraday-org/chartlang` and resolve (link-check if the docs
  gate includes one).
- README line-limit gate (≤ 100) must hold for all five after templating.
