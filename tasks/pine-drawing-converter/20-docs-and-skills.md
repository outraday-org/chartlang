# Task 20 — Documentation + skill updates

> **Status: TODO**

## Goal

Ship user-facing documentation for the Pine v6 → chartlang converter:
a docs section under `docs/converter/` covering install / usage /
diagnostics / supported surface / hard-rejects, the per-diagnostic
reference pages, and an update to `skills/chartlang-coding/` adding a
"converting from Pine" reference page that authors can consult when
porting scripts by hand or when they hit a converter reject. Update
the package README to point at the docs and add a section to the root
README announcing the converter.

## Prerequisites

Task 19 (the full pipeline is exercised by fixtures, so the docs can
cite real conversion examples).

## Current Behavior

`packages/pine-converter/README.md` is a §22.4 template stub. No
`docs/converter/` exists. `skills/chartlang-coding/` has no Pine
conversion reference. Root README does not mention the converter.

## Desired Behavior

- `docs/converter/index.md` — Landing page with intro + quick-start.
- `docs/converter/usage.md` — CLI and programmatic usage.
- `docs/converter/supported.md` — Supported Pine surface (drawings,
  inputs, control flow, partial ta/math, request.security partial).
- `docs/converter/rejects.md` — The hard-reject table with each
  diagnostic code, its meaning, and the recommended manual rewrite.
- `docs/converter/diagnostics/` — One short page per diagnostic code
  (the URL `docs:converter/diagnostics/<code>` linked from the CLI's
  formatter — see Task 17 §7).
- `docs/converter/examples/` — Side-by-side Pine + chartlang
  conversion examples generated from the fixture corpus (Task 19).
- `skills/chartlang-coding/references/translating-from-pine.md` —
  A short reference page authors can read when porting scripts by
  hand or interpreting converter diagnostics.
- `packages/pine-converter/README.md` — Updated to point at the docs.
- Root `README.md` — One paragraph + link under the existing
  feature/tooling list.

## Requirements

### 1. `docs/converter/index.md`

Sections:
- **What it does** — one paragraph: "Convert Pine Script v6 indicators
  to chartlang `.chart.ts` scripts. Drawings v1 — covers `line.new`,
  `label.new`, `box.new`, `table.new`, `polyline.new`, `linefill.new`,
  plus inputs, control flow, and partial `ta.*`/`math.*` passthrough."
- **Status** — `@experimental`. Drawings v1; future slices to
  extend.
- **Quick start** — 6-line CLI invocation + 4-line programmatic
  invocation.
- **What you get** — bullet list: deterministic source-to-source,
  compiles through chartlang-compiler, structured diagnostics with
  source spans, three camp strategies for drawings.
- **What it won't do** — strategies (downgrade to indicator),
  matrices/maps/UDT, dynamic-collection drawings (with link to
  rejects).

### 2. `docs/converter/usage.md`

CLI section (full `--help` output) + programmatic section
(`convert`, `convertFile`, `ConvertOpts`, `ConvertResult`). Includes
a Node + ESM snippet and a deno-compat note.

### 3. `docs/converter/supported.md`

A table per pillar:

- **Drawings**: every chartlang `draw.*` kind reachable, with the
  Pine construct that maps to it.
- **Inputs**: every Pine `input.*` form and the chartlang `input.*`
  it maps to.
- **Control flow**: which Pine constructs convert (literal-`for`,
  `if`/`else`, `switch`, ternary) and which don't (`while`,
  `for...in`, `for...of` — and what to do instead).
- **ta.* / math.* / str.***: the rough passthrough subset + how to
  read "not mapped" warnings.

### 4. `docs/converter/rejects.md`

The hard-reject catalogue (from Tasks 12, 14, 15). For each: code,
when it fires, why it's a reject (the architectural reason), and the
recommended manual rewrite. Generated semi-automatically from
`DIAGNOSTIC_CODES` (Task 17) — write a small `scripts/gen-converter-
docs.ts` script that reads the codes registry + per-code body markdown
and emits the page.

### 5. `docs/converter/diagnostics/<code>.md`

One file per diagnostic code. ~10 lines each: code, severity, when it
fires, example Pine snippet, example output (or non-output, for
rejects). The CLI links to `https://chartlang.dev/converter/
diagnostics/<code>` per Task 17 §7. The body of each file is
hand-written but file existence + frontmatter is generated by the
same `gen-converter-docs.ts` script.

### 6. `docs/converter/examples/`

For each fixture in the Task 19 corpus, generate a side-by-side page:
- The fixture's Pine source.
- The expected chartlang output.
- The diagnostic list (collapsed by default).
- A "view runtime trace" link (the conformance scenario's golden
  emissions).

Generation: a `scripts/gen-converter-examples.ts` script reads each
fixture triple and emits the page. Append to `pnpm` scripts as
`pnpm converter:examples:generate` + `--check` mode for the docs
gate.

### 7. `skills/chartlang-coding/references/translating-from-pine.md`

A focused reference for script authors (not implementers):

- "Why might my Pine script not convert?" — Quick guide to reading
  converter diagnostics.
- The three drawing camps explained from the author's perspective:
  "If your script does X, write it like Y."
- The hard-reject patterns with recommended Pine rewrites: "Use
  `array<line>` with `max_lines_count` + `array.shift` eviction to
  trigger Camp B."
- Common gotchas: `varip` semantics, future bar_index anchors,
  `yloc.abovebar` padding.
- Link list to the docs/converter/ pages.

Keep ≤300 lines. Cap the page at "what authors need to know" —
implementer detail belongs in `docs/converter/`.

`docs/getting-started/*.md`. Blocks under `docs/converter/*.md`
are **not** compiled. Either:
1. Tag any unsupported snippets `\`\`\`ts no-gate`, or
