## Decision: `request-security-different-symbol` becomes an INFO

Per `pine-converter/CLAUDE.md` "code STRINGS never change" — the rule forbids
RENAMING a code, not changing its SEVERITY/MESSAGE. So:

- A **mappable literal symbol** (`stringLiteralValue` ≠ null): lower to
  `{ symbol: <literal>, interval: <interval> }` and emit
  `request-security-different-symbol` as an **info** ("cross-symbol read mapped
  to chartlang multi-symbol") so downstream tooling still sees the cross-symbol
  signal but it does NOT block compile (info ≠ error → `fixtures-compile`
  round-trips). The CODE STRING is unchanged.
- `syminfo.tickerid`: byte-identical to today — `{ interval }`, no symbol, no
  diagnostic.
- Anything else (computed/non-literal symbol): push `request-security-not-mapped`
  (existing error) and return `null` (un-mappable — chartlang needs a literal
  symbol too).

This is the minimal-diff option (no new code), keeps the registry append-only,
and keeps the round-trip green. `converter:docs:check` regenerates
`docs/converter/diagnostics.md` from the registry, so the severity/message
change flows through it — must re-run `pnpm converter:docs:generate`.
