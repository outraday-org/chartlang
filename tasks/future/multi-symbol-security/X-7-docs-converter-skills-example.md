## Desired Behavior

- A different-symbol Pine `request.security("NASDAQ:AAPL", "1D", close)`
  lowers to `request.security({ symbol: "NASDAQ:AAPL", interval: "1D" }).close`
  (data form) / `request.security({ symbol: …, interval: … }, (bar) => …)`
  (expression form) — the symbol is **carried into the opts**, not dropped.
  `request-security-different-symbol` is **retired** (or downgraded to an info
  noting the cross-symbol mapping) for a *mappable literal* symbol; a genuinely
  un-mappable symbol (a non-literal `syminfo.tickerid` expression, a computed
  ticker) still warns via `request-security-not-mapped`.
- The multi-timeframe guide + `request.security` primitive page document the
  `{ symbol, interval }` form, the `multiSymbol` capability + NaN fallback, and
  the literal-symbol constraint.
- `translating-from-pine.md` gains a `request.security(symbol, …)` row.
- `primitives.md` regenerated; `skills:gate` green.
- A `symbol-ratio.chart.ts` example (SPY/QQQ ratio) + a DEMO_SCRIPTS entry.

## Requirements

### 1. Converter: carry the symbol into the opts (`requestSecurity.ts`)

- Resolve the symbol arg: if it is `syminfo.tickerid` → emit `{ interval }`
  (chart symbol, omit `symbol` — byte-identical to today's same-symbol
  output). If it is a **string literal** (`"NASDAQ:AAPL"`, mapped via
  `stringLiteralValue`) → emit `{ symbol: <literal>, interval: <interval> }`.
  Anything else (a computed symbol, a non-`tickerid` identifier expression) →
  push `request-security-not-mapped` and return `null` (un-mappable), since the
  chartlang side requires a literal symbol too.
- Build the opts string with the symbol slot:
  `{ symbol: ${JSON.stringify(symbol)}, interval: ${JSON.stringify(interval)} }`
  (symbol omitted for the `tickerid` case). Keep the data-vs-callback split
  (line 171–173) unchanged otherwise.
- Retire / repurpose `request-security-different-symbol`: per
  `packages/pine-converter/CLAUDE.md` "code strings are the stable public
  contract and never change" — do **not** delete the code string; instead stop
  emitting it for a mappable literal symbol, and (optionally) emit it as an
  **info** "cross-symbol read mapped to chartlang multi-symbol" when a
  non-chart symbol is lowered, so downstream tooling still sees the
  cross-symbol signal. Decide and document which; keep the code registered.
- Update the converter unit tests (`request-security.test.ts`): a literal
  different symbol now produces the symbol-bearing opts (no longer a warning);
  `syminfo.tickerid` is byte-identical to today; a computed symbol →
  `request-security-not-mapped`.
